Command Lines¶

Process Composition¶

Jubatus is client-server configuration. The simplest configuration is single client process and single server process like following figure.

Jubatus can make use of multiple server processes in order to scale out the system. By throwing queries to jubakeeper, we can execute train/classify with multiple server processes as if clients are interacting with a single server. jubakeeper does alive monitoring and load balancing with Zookeeper.

Sometimes we want to distribute clients because data size is huge, or source of data is far from server. Jubatus can achieve this by creating multiple jubakeepers and assigning different jubakeeper to each client.

Online Machine Learning Algorithms¶

jubaclassifier¶

-p, --rpc-port¶

Port number for RPC([= 9199])

-c, --thread¶

The number of threads RPC can handle similtaneously([= 2])

-t, --timeout¶

Session timeout of RPC in second([= 10])

-z, --zookeeper¶

Server and port number of Zookeeper. If this option is not specified, Zookeeper runs standalone mode. This option must be specified unless --storage option is local. Format of this option must be ipaddress:port,hostname:port,... . No spaces between values are allowed.

-n, --name¶

Instance name, which is unique among zookeeper cluster specified with --zookeeper . If local_mixture is specified in --storage , parameters are mixed among all instances whose names are identical. This option must not contain characters which are not available as znode (such as ‘/’).

-d, --tmpdir([=/tmp])¶

save Location where training model is saved when “save” API is published. Default value is /tmp . load Location from where training model is loaded when “load” API is publised. Default value is /tmp .

-l, --logdir¶

Location of logfiles. If this option is not specified, log messages output to stderr.

-j, --join¶

Join to the existing clister. New processes should not join to the existing cluster without specifying this option otherwise the machine learning won’t work.

-s, --interval_sec([=16])¶

Interval time of one of ‘’mix’’ trigger in seconds.

-i, --interval_count([=512])¶

Another ‘’mix’’ trigger: By default, in each 512 update requests (in single server) mix is tried to be invoked.

Send specified command to registered to Zookeeper. Command should be one of the followings.

start start N jubaclassifiers with option –name=NAME –storage=STORAGE –zookeeper=ZOOKEEPER. N, NAME, STORAGE and ZOOKEEPER are the ones specified with --name , --storage , --zookeeper , and --zookeeper , respectively.

tools¶

jubaconv¶

jubaconv is a debug tool which converts specified data is converted in a scecified format. It simulates internal behavior of fv_converter in the console. We can utilize this in order to check if converter_config works correctly.

-i (json/datum)¶

Specifies input format.

-o (json/datum/fv)¶

Specifies output format. Processing is simulated until specified format is obtained.

-c converter_config¶

Specifies converter_config.

jenerator¶

jenerator is a converter to create a jubakeeper and basic server template from extended msgpack-idl format. jenerator takes first argument as an IDL filename to process, rest is for options.

-t¶

Generates server template.

-i¶

Generates the code as internal library to merged in Jubatus repository. If the generated code is separated from Jubatus repository and assumed Jubatus is installed, or you don’t know well, DO NOT use this option.

-o path¶

(NOT IMPLEMENTED) Directory to output the generate source code files.
